  - The route uses `getSupabaseAdminClient()` to read categories and generate a template, but the route handler itself does not call `requireAdmin()` or `ensureAdminApiAccess()`.
  - Middleware protects `/admin`, but service-role route handlers should still enforce authorization at the handler boundary for defense in depth and parity with `parseAdminPartnerXlsxFileAction`.

### Medium

- [ ] Track the residual Next-bundled `postcss` audit advisory.
  - After upgrading to the current latest `next@16.2.6`, `npm audit --audit-level=moderate` still reports `next/node_modules/postcss <8.5.10`.
  - `npm audit fix --force` is not acceptable because it attempts to install `next@9.3.3`, which would be a breaking downgrade.
  - Re-run audit after the next patched Next release is available, then upgrade normally.

- [ ] Add abuse controls for product/event logging endpoints.
  - File: `src/app/api/events/product/route.ts`
  - The route validates event names and same-origin access, then logs events asynchronously, but it has no IP/session rate limit.
  - This is also a performance concern: `public.event_logs` inserts dominate DB time in recent query stats.
  - Add rate limiting, sampling, or server-side debounce for low-value repeated events. Logging failures must not block user-visible requests.

- [ ] Standardize CSRF protection for cookie-authenticated mutations.
  - Several routes use local same-origin checks, but the checks are not centralized and often tolerate missing `Origin`.
  - Add a shared helper for state-changing route handlers that validates `Origin` or trusted `Referer`, method, and expected content type. Document explicit exceptions.
  - Apply it consistently to user, partner, push, review, and admin mutation routes that rely on cookies.

- [ ] Harden direct review media uploads.
  - File: `src/app/api/partners/[id]/reviews/media/upload-url/route.ts`
  - The signed upload flow validates client-declared MIME type and size before issuing signed URLs.
  - Add server-side post-upload verification or cleanup for unused/invalid objects, and consider content sniffing for review media.

- [ ] Align middleware session-secret handling with server-side fail-closed behavior.
  - File: `middleware.ts`
  - Admin middleware already rejects missing/short `ADMIN_SESSION_SECRET`.
  - User and partner middleware currently fall back to an empty string when secrets are missing, while server helpers throw on missing/short secrets.
  - Make middleware reject user/partner session verification when the relevant secret is missing or too short.

- [ ] Create a service-role access boundary inventory.
  - The app intentionally uses Supabase service role in server-only modules because custom cookie auth and repository patterns are used.
  - Add a lightweight guard/wrapper convention or lintable import rule so service-role reads/writes only happen in server modules behind an explicit user/admin/partner context.
  - Include route handlers, server actions, cron jobs, and repositories in the inventory.

### Low

- [ ] Revisit admin authentication hardening.
  - Current admin auth uses env-based credentials, signed httpOnly cookies, rate limiting, IP/basic-auth optional guards, and timing-safe password comparison.
  - Plan a stronger operator auth path, such as external IdP/MFA, before broader admin usage.

- [ ] Use timing-safe comparison for optional admin basic auth.
  - File: `src/lib/admin-security.ts`
  - Basic auth is an optional extra gate, but matching it with the admin password comparison pattern would remove a small timing side channel.

- [ ] Add an automated RLS/revoke regression check.
  - `supabase/schema.sql` enables RLS and revokes broad access on many sensitive tables.
  - Add a schema test that fails if sensitive tables lose RLS or regain direct `anon`/`authenticated` access.

## Positive Findings

- Admin, user, and partner sessions use signed httpOnly cookies with `sameSite: "lax"` and `secure` in production.
- Admin login has account/IP throttling and timing-safe password verification.
- Major sensitive tables have RLS enabled and broad `anon`/`authenticated` grants revoked in the schema snapshot.
- Review and favorite APIs require a signed member session and enforce ownership where needed.
- The suggestion API validates payloads, escapes HTML, and has IP-based rate limiting.
- The image proxy blocks private/reserved IP ranges, rejects non-image content types, caps response size, and uses request timeouts.
- Cron endpoints require either an admin session or `CRON_SECRET`.
