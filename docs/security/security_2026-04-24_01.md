### [ ] HIGH — 실제 외부 발송이 성공한 뒤 후처리 저장이 실패하면 실패로 재분류되어 중복 알림을 만들 수 있다

- 파일: [src/lib/admin-notification-ops-delivery.ts](/Users/myknow/coding/ssartnership/src/lib/admin-notification-ops-delivery.ts:58)
- 관련 라인: 58-113, 170-203

현재 푸시와 Mattermost 발송 로직은 “실제 외부 발송”과 “발송 후 DB/로그 기록”을 하나의 `try` 블록 안에서 같이 처리한다. 이 구조에서는 실제 사용자에게 알림이 정상 발송된 뒤, 후속 bookkeeping이 실패해도 전체가 실패로 떨어진다.

구체적으로:
- 푸시 발송은 `webpush.sendNotification(...)`이 성공한 뒤 `markPushSuccess`, `logPushDelivery`, `notificationRepository.recordNotificationDelivery`를 수행한다.
- 이 중 하나라도 실패하면 `catch`로 들어가 동일 발송을 실패로 기록한다.
- Mattermost는 더 심각하다. `sendPost(...)`가 이미 성공했는데 후속 저장이 실패하면, 다음 sender candidate year로 재시도하면서 같은 수신자에게 중복 메시지를 보낼 수 있다.

영향:
- 사용자에게 실제로는 성공적으로 도달한 알림이 시스템상 실패로 남을 수 있다.
- 운영자는 실패로 오인해 재시도할 수 있고, 그 결과 중복 푸시/중복 Mattermost 메시지가 발생할 수 있다.
- 일시적인 DB 장애가 외부 사용자 대상 중복 알림으로 확대된다.

권장 조치:
- “외부 발송”과 “발송 결과 기록”을 분리해야 한다.
- 외부 발송이 성공한 뒤 후속 저장이 실패하면, 상태를 `sent_with_bookkeeping_error` 같은 별도 상태로 남기고 재발송 루프로 보내지 않아야 한다.
- Mattermost sender fallback은 `sendPost(...)` 이전 단계에서만 재시도하고, 발송 성공 이후에는 절대로 다른 sender로 넘기지 않아야 한다.

### [ ] HIGH — 캠페인 전체 발송이 끝난 뒤 metadata 업데이트 실패만으로 전체 요청이 실패처럼 보일 수 있다

- 파일: [src/lib/admin-notification-ops.ts](/Users/myknow/coding/ssartnership/src/lib/admin-notification-ops.ts:647)
- 관련 라인: 647-659

`sendAdminNotificationCampaign`은 채널별 발송을 모두 완료한 다음, 마지막에 `notificationRepository.updateNotificationMetadata(...)`를 호출한다. 이 마지막 metadata 업데이트가 실패하면 함수 전체가 reject된다.

영향:
- 실제 푸시/Mattermost/in-app 발송은 이미 끝났는데, 호출자 입장에서는 전체 요청이 실패한 것처럼 보인다.
- 운영자가 “실패했으니 다시 보내야 한다”고 판단하면 동일 대상에게 캠페인이 한 번 더 발송될 수 있다.
- 즉, 최종 bookkeeping 실패가 전체 브로드캐스트 중복 전송으로 이어질 수 있다.

권장 조치:
- 채널 발송이 끝난 뒤 metadata 업데이트 실패는 별도의 후처리 실패로 취급해야 한다.
- 최종 반환값은 발송 결과를 우선 보존하고, metadata 저장 실패는 경고/보조 상태로 분리해야 한다.
- 가능하면 발송 idempotency key 또는 campaign dedupe key를 추가해 운영자 재시도 시 중복 발송을 막는 것이 좋다.

`verifyAdminToken()`은 `getAdminSecret()`으로 서명을 검증하는데, 현재 `getAdminSecret()`은 `ADMIN_SESSION_SECRET`가 없을 때 빈 문자열(`""`)로 대체된다. 이는 [src/lib/auth.ts](/Users/myknow/coding/ssartnership/src/lib/auth.ts:12)에서 secret이 없거나 너무 짧으면 예외를 던지고 즉시 실패 처리하는 방식과 다르다.

영향:

- 환경 설정이 잘못된 경우, middleware가 빈 secret 기준으로 서명된 토큰을 통과시킬 수 있고, 이후 레이아웃/API 보호 로직에서 다시 막히기 전까지 요청이 더 깊이 들어갈 수 있다.

권장 조치:
- middleware도 `src/lib/auth.ts`와 동일한 secret 검증 규칙을 사용하도록 맞춘다.
- `ADMIN_SESSION_SECRET`가 없거나 너무 짧으면 빈 문자열로 검증하지 말고 즉시 거부해야 한다.

### [ ] MEDIUM — 로그 로더가 일부 페이지를 조용히 누락시켜도 정상 결과처럼 보일 수 있다

- 파일: [src/lib/log-insights/data.ts](/Users/myknow/coding/ssartnership/src/lib/log-insights/data.ts:21)
- 관련 라인: 21-71

`queryAllRows()`는 이제 로그 페이지를 병렬로 가져온다. 이 과정에서 특정 페이지 조회가 실패하면 에러를 기록하고 빈 결과로 대체한다.

- `return { rows: [] as T[], error: true };`
- 이후 `if (pageResult.error) { continue; }`

즉, 호출자는 일부 페이지가 비어 있는 부분 결과를 받지만, 현재는 그 사실을 UI에 드러내는 별도 상태가 없다. 기존 `truncated`는 행 개수 상한을 의미할 뿐, 조회 실패로 인한 누락을 의미하지 않는다.

영향:
- 감사/보안 조사 시 일부 로그가 빠졌는데도 화면상으로는 정상처럼 보일 수 있다.
- 특히 `/admin/logs`는 사고 대응이나 운영 조사에 쓰일 가능성이 있어, 부분 누락을 숨기면 위험하다.

권장 조치:
- `truncated`와 별도로 `partialFailure` 같은 플래그를 추가해 UI에 표시한다.
- 페이지 일부 조회 실패가 발생하면 화면에서 명확한 경고를 노출한다.
- 무결성이 더 중요한 화면이라면, 순차 조회로 fallback하거나 아예 non-200 응답으로 실패를 드러내는 것도 고려해야 한다.

### [ ] MEDIUM — 병렬 로그 fan-out이 넓은 조회 범위와 export에서 읽기 스파이크를 만들 수 있다

- 파일: [src/lib/log-insights/data.ts](/Users/myknow/coding/ssartnership/src/lib/log-insights/data.ts:29)
- 관련 라인: 29-53, 155-186

`queryAllRows()`는 현재 페이지 오프셋을 모두 미리 계산한 뒤 `Promise.all()`로 한 번에 조회한다. CSV export에서는 그룹별 최대 20개 페이지를 동시에 읽을 수 있어, 3개 로그 테이블 기준으로 최대 60개의 동시 read가 발생할 수 있다.

영향:
- 넓은 범위 export나 대량 로그 조회에서 DB/read burst가 생길 수 있다.
- 인증 우회 같은 직접 보안 문제는 아니지만, 운영/가용성 측면에서는 유의미한 리스크다.

권장 조치:
- 전체 페이지를 한 번에 던지지 말고 bounded concurrency를 둔다.
- 짧은 페이지가 나오면 이후 조회를 중단하도록 구조를 바꾼다.
- export 모드는 순차 혹은 소규모 병렬 풀 방식이 더 안전하다.
